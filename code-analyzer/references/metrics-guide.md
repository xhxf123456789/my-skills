## 圈复杂度

### 定义
圈复杂度（Cyclomatic Complexity）由 Thomas McCabe 于 1976 年提出，用于衡量代码的控制流复杂度。

### 计算公式
```
CC = E - N + 2P
```
其中：
- E = 控制流图中的边数
- N = 控制流图中的节点数
- P = 连接组件数（对于单个函数，P=1）

### 简化计算
对于单个函数，圈复杂度可以通过以下方式计算：
```
CC = 决策点数量 + 1
```

### 决策点类型
| 决策点 | 示例 | 复杂度贡献 |
|-------|------|-----------|
| if 语句 | if condition: | +1 |
| elif 语句 | elif condition: | +1 |
| for 循环 | for item in list: | +1 |
| while 循环 | while condition: | +1 |
| except 处理 | except Exception: | +1 |
| with 语句 | with open(): | +1 |
| assert 语句 | assert condition | +1 |
| 条件表达式 | x if cond else y | +1 |
| 逻辑运算符 | and, or | +1 |

### 示例代码
```python
def calculate_grade(score):
    if score >= 90:          # +1
        return 'A'
    elif score >= 80:        # +1
        return 'B'
    elif score >= 70:        # +1
        return 'C'
    elif score >= 60:        # +1
        return 'D'
    else:
        return 'F'
# 圈复杂度 = 4 (if/elif) + 1 = 5
```

### 等级标准
| 等级 | 复杂度范围 | 风险等级 | 建议 |
|-----|-----------|---------|------|
| A | 1-5 | 简单 | 容易理解和测试 |
| B | 6-10 | 中等 | 需要一定测试成本 |
| C | 11-20 | 复杂 | 测试困难，建议重构 |
| D | 21-30 | 很复杂 | 测试非常困难，必须重构 |
| E | 31-40 | 极复杂 | 不可测试，立即重构 |
| F | 41+ | 不可接受 | 必须重写 |

## 认知复杂度

### 定义
认知复杂度（Cognitive Complexity）由 SonarSource 提出，更贴近人类理解代码的难度。

### 核心原则
1. **忽略简化结构**：不增加简单结构的复杂度
2. **打断线性流**：增加理解难度
3. **嵌套惩罚**：嵌套结构增加额外复杂度

### 计算规则
| 结构 | 复杂度 | 说明 |
|-----|-------|------|
| if | +1 | 打断线性流 |
| elif | +1 | 打断线性流 |
| else | +0 | 不打断线性流 |
| for | +1 | 循环结构 |
| while | +1 | 循环结构 |
| catch | +1 | 异常处理 |
| switch case | +1 | 分支结构 |
| 嵌套结构 | +嵌套深度 | 额外惩罚 |
| break/continue | +1 | 跳出流程 |
| 逻辑运算符 | +1 | and/or |

### 与圈复杂度的区别
| 特性 | 圈复杂度 | 认知复杂度 |
|-----|---------|-----------|
| 关注点 | 控制流 | 理解难度 |
| 线性代码 | 计数 | 不计数 |
| 嵌套结构 | 不额外计数 | 额外惩罚 |
| else 语句 | 计数 | 不计数 |

### 示例对比
```python
# 圈复杂度 = 3，认知复杂度 = 5（包含嵌套惩罚）
def process_data(data):
    if data:                  # +1（圈），+1（认知）
        for item in data:     # +1（圈），+1（认知）+1（嵌套）
            if item.valid:    # +1（圈），+1（认知）+1（嵌套）
                process(item)
```

## Halstead 指标

### 定义
Halstead 指标基于代码中的运算符和操作数统计，用于衡量代码复杂度和工作量。

### 基础度量
| 符号 | 名称 | 说明 |
|-----|------|------|
| n1 | 不同运算符数 | 如 +, -, if, for |
| n2 | 不同操作数数 | 如变量名、常量 |
| N1 | 运算符总数 | 所有运算符出现次数 |
| N2 | 操作数总数 | 所有操作数出现次数 |

### 导出度量
| 指标 | 公式 | 说明 |
|-----|------|------|
| 程序长度 | N = N1 + N2 | 代码总长度 |
| 程序词汇量 | n = n1 + n2 | 不同符号数量 |
| 程序体积 | V = N × log2(n) | 信息量 |
| 程序难度 | D = (n1/2) × (N2/n2) | 实现难度 |
| 编程工作量 | E = D × V | 开发工作量 |
| 预测错误数 | B = V / 3000 | 潜在缺陷数 |

## 维护性指数

### 定义
维护性指数（Maintainability Index）是综合指标，范围 0-100，越高表示越易维护。

### 计算公式
```
MI = 171 - 5.2 × ln(V) - 0.23 × CC - 16.2 × ln(LOC)
```
其中：
- V = Halstead 体积
- CC = 圈复杂度
- LOC = 代码行数

### 等级标准
| 等级 | 指数范围 | 说明 |
|-----|---------|------|
| 高 | 20-100 | 易于维护 |
| 中 | 10-19 | 中等难度 |
| 低 | 0-9 | 难以维护 |

## 实践应用

### 复杂度控制策略
1. **设定阈值**：团队统一复杂度上限（如 10）
2. **代码审查**：关注高复杂度函数
3. **持续监控**：集成到 CI/CD 流程
4. **逐步重构**：优先处理高复杂度代码

### 重构技巧
| 问题 | 重构方法 | 效果 |
|-----|---------|------|
| 深层嵌套 | 提取方法、提前返回 | 降低认知复杂度 |
| 多分支逻辑 | 策略模式、状态模式 | 降低圈复杂度 |
| 长函数 | 提取方法、分解函数 | 降低总体复杂度 |
| 复杂条件 | 提取条件表达式 | 提高可读性 |

# radon 复杂度分析

### 持续改进
1. **定期评估**：每月进行复杂度分析
2. **趋势跟踪**：监控复杂度变化趋势
3. **目标设定**：制定复杂度降低目标
4. **知识分享**：团队分享重构经验
